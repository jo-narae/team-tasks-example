# team-tasks 요건 정의

> 출처: `docs/personas.md` (페르소나 3종 · 9개 시나리오), `docs/user-stories.md` (민호 하루 타임라인).
> 범위: MVP 헌장 — 단일 테이블 `tasks(id, title, description, assignee, status, created_at)`, 기능 5개 이하, 1일 구현. **Phase 2 제안 없음.**

## 1. 기능 요건

| ID | 이름 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| F1 | 일감 생성 | 제목 + 담당자(기본값: 본인)만으로 일감을 즉시 등록한다. 설명은 비워둘 수 있다. (민호 14:00 / 지수 시나리오 1) | P0 |
| F2 | 목록 조회 + 담당자 필터 | 전체 일감을 상태별로 묶어 보여주고, "내 일감(담당자 = 본인)" 필터를 토글할 수 있다. 세 페르소나 전원의 진입 화면. (민호 09:00 / 서연 시나리오 1) | P0 |
| F3 | 상태 토글 | 일감의 상태를 `todo → doing → done` 사이에서 한 번에 전환한다. 전환 즉시 다른 사용자 화면에 반영될 필요는 없고, 새로고침 기준으로 정합하면 된다. (민호 09:15·12:30 / 민호 시나리오 2) | P0 |
| F4 | 설명 편집 | 일감의 설명 필드를 누구나 인플레이스로 수정한다. 의도 메모·대기 사유 기록용. (민호 11:00·17:00 / 민호 시나리오 3) | P1 |
| F5 | 담당자 변경 | 일감의 담당자를 다른 팀원으로 재할당한다. 과부하 분산 및 잘못된 배정 정정 용도. (지수 시나리오 2) | P1 |

> 합계 5개 — MVP 헌장의 "기능 5개 이하" 한도와 일치.
> P0 3개는 1일 구현 마감 안에 반드시 들어가야 하고, P1 2개는 같은 날 안에 마감을 위협하면 다음 영업일로 넘긴다.

## 2. 비기능 요건

각 영역에 측정 가능한 수치 1~2개씩만 둔다. 모두 단일 region · 단일 테이블 가정.

### 2.1 성능
- 목록 API(`GET /api/tasks`, 100건 기준) **p95 응답 200ms 이내**.
- `/tasks` 첫 화면 **LCP 1.5s 이내** (사내 LAN, 데스크톱 Chrome 기준).

### 2.2 보안
- 인증된 사내 사용자만 `/api/tasks` 의 모든 메서드 호출 가능 — **비인증 요청은 401 응답률 100%**.
- 입력 검증: `title` **1~120자**, `description` **0~2000자**, `status ∈ {todo, doing, done}`. 위반 시 400.

### 2.3 가용성
- 업무시간(평일 09:00~18:00 KST) **월간 가용성 99.5%** (월 다운타임 약 22분 이하).
- 배포로 인한 계획 다운타임은 **업무시간 외에만**, 1회 5분 이내.

### 2.4 접근성
- 모든 핵심 동작(F1~F5) **키보드만으로 수행 가능** — 마우스 의존 0건, axe-core 위반 0건.
- 본문 텍스트와 배경 색상 대비 **WCAG 2.1 AA(4.5:1) 이상**.
