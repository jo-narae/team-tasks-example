# team-tasks 사용자 스토리

> 대상 페르소나: **실무 담당자 민호 (3년 차 개발자, 일반 직원)** — `docs/personas.md` 참조.
> 사용 기능 범위: MVP 헌장 5개 기능(일감 생성 / 목록 조회·필터 / 담당자 변경 / 상태 토글 / 설명 편집) 안에서만 진행.

## 1. 핵심 시나리오

하루 동안 민호가 `/tasks` 화면을 어떻게 쓰는지 시간 순서대로 정리한다.

### 09:00 — 출근, 오늘 할 일 파악
- 자리에 앉아 노트북을 열고 사내 포털에서 `/tasks` 로 들어간다.
- 기본 화면은 전체 목록이라 한 번 클릭으로 **"내 일감(담당자 = 민호)" 필터**를 켠다.
- 상태별로 묶인 목록에서 `todo` 3건, `doing` 2건이 남아 있는 걸 확인한다. 모바일이 아닌 데스크톱에서 보지만, 같은 화면이라 추가 학습 없이 바로 읽힌다.

### 09:15 — 첫 일감 시작
- 우선순위가 높아 보이는 `todo` 한 건("로그인 API 에러 메시지 통일")을 클릭한다.
- 설명 필드를 읽고 작업 의도를 확인한 뒤, 상태를 **`todo` → `doing`** 으로 토글한다. 이 토글이 리드 지수에게는 "민호가 이걸 잡았다"는 신호가 된다.

### 11:00 — 모호한 일감에 메모 남기기
- 다음 `todo`("회원가입 폼 정리")를 열었더니 설명이 한 줄뿐이라 무엇을 정리해야 할지 모호하다.
- 리드에게 다시 묻기 전에, 본인이 추측한 작업 범위를 **설명 필드에 한 줄 추가**한다. 예: "비밀번호 규칙 안내 문구만 교체하는 걸로 이해함, 다르면 코멘트 부탁."
- 상태는 아직 `todo` 로 두고, 리드가 지나가다 볼 수 있게 한다.

### 12:30 — 점심 전, 진행 중 일감 마무리
- 오전에 작업한 "로그인 API" 일감의 PR 을 올린 뒤, 화면으로 돌아와 상태를 **`doing` → `done`** 으로 한 번에 토글한다.
- "내 일감" 필터를 유지한 채 남은 항목 수가 줄어드는 걸 확인하고 점심을 먹으러 간다.

### 14:00 — 점심 후, 새 일감 추가
- 점심 자리에서 동료가 "에러 로그가 한국어/영어 섞여 나온다"고 흘리듯 말했다. 잊지 않으려고 자리에 돌아오자마자 본인 이름으로 **새 일감을 즉시 등록**한다 — 제목만 적고 담당자는 자기 자신, 상태 `todo`.
- 굳이 설명을 채우지 않는다. 본인 머릿속에 맥락이 있는 동안엔 제목만으로 충분하다.

### 15:30 — 잡혀 있던 일감 본격 진행
- 오후 집중 시간대에 "회원가입 폼 정리" 일감을 다시 열고, 오전에 본인이 추가한 메모대로 작업을 시작한다.
- 상태를 **`doing`** 으로 바꾼 뒤 코드를 만진다.

### 17:00 — 막힌 일감 표시
- "회원가입 폼 정리"는 절반쯤 진행했지만, 추가로 디자인 확인이 필요해 오늘 안에 끝나지 않는다.
- 상태는 **`doing` 그대로 둔다**. 신규 입사자 서연의 시나리오와 같은 의도 — "지금 잡고는 있지만 멈춰 있다"는 신호를 리드가 알아채게 하기 위함.
- 설명 필드에 "디자인 시안 확정 대기"를 한 줄 덧붙여 둔다.

### 17:45 — 퇴근 전 정리
- "내 일감" 필터에서 오늘 `done` 으로 바뀐 항목을 훑어보며 마무리 회고를 한다.
- 내일 가장 먼저 잡을 `todo` 한 건을 시각적으로 머릿속에 새겨 두고 노트북을 닫는다.

### 18:00 — 퇴근
- 별도의 "내 작업 보고" 화면이나 일일 리포트 작성 없이 퇴근. 리드는 같은 `/tasks` 를 보고 민호의 하루를 그대로 따라갈 수 있다.

---

#### 이 시나리오에서 사용된 MVP 기능 매핑
| 시각 | 동작 | 사용 기능 |
| --- | --- | --- |
| 09:00 | "내 일감" 필터 | ② 목록 조회(담당자 필터) |
| 09:15 / 12:30 / 15:30 | 상태 변경 | ④ 상태 토글 |
| 11:00 / 17:00 | 의도/대기 사유 메모 | ⑤ 설명 편집 |
| 14:00 | 점심 후 새 일감 추가 | ① 일감 생성 |

담당자 변경(③)은 민호 본인 시나리오 안에서는 발생하지 않는다 — 그건 리드 지수의 시나리오에서 다룬다.

## 2. 결정 사항

위 시나리오를 구현으로 옮기기 전에 답해야 할 5가지 — 모두 단일 테이블 + 기능 5개 헌장 안에서 풀 수 있어야 한다.

1. **담당자 변경 권한은 누구에게 있는가?** 본인이 본인 일감의 담당자를 다른 팀원으로 바꿀 수 있는가, 아니면 리드만 가능한가? (시나리오에서 민호는 본인 외 사람에게 배정하지 않지만, 14:00 새 일감을 잘못된 담당자에게 넘겨야 할 때 이 결정이 필요해진다.)
2. **상태값은 `todo / doing / done` 3단계로 충분한가?** "막힘(blocked)" 같은 4번째 상태가 필요한가, 아니면 17:00 사례처럼 `doing` + 설명 메모로 대체하는 게 헌장에 더 맞는가?
3. **설명 필드는 누구나 편집 가능한가?** 11:00 민호처럼 담당자가 본인 일감의 설명을 자유롭게 고쳐도 되는가, 아니면 작성자(리드)만 가능하고 담당자는 별도의 코멘트 수단이 있어야 하는가? (후자라면 "단일 테이블"을 깨야 한다.)
4. **"내 일감" 필터는 로그인 시 기본 적용인가?** 09:00 민호가 매번 필터를 켜야 하는가, 아니면 로그인 사용자에게는 기본이 "내 일감"이고 전체 보기는 토글인가?
5. **완료된 일감은 목록에서 어떻게 다루는가?** 12:30 `done` 토글 직후 해당 항목이 목록에서 자동으로 사라지는가(아카이브), 아니면 끝까지 보이고 17:45 회고용 기본 표시로 유지되는가?
